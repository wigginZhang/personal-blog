# **OpenClaw 部署心得：从理想走向现实**

4月8日，我花了一些时间将 **OpenClaw** 部署在我的笔记本上，作为一个小型的服务器使用。这个过程让我对工具的部署和使用有了更深的理解，也让我意识到一些可以改进的地方。今天，我想和大家分享一下我的部署心得。

## **部署环境与配置**

首先，我选择将 **OpenClaw** 部署在我的笔记本上，作为服务器使用。为了方便公司电脑的访问，我在公司电脑上配置了企业微信的 **channel** 作为入口。这样一来，我可以通过在 **channel** 上输入指令，让 **小o** 配置 **search skill**。

### **配置过程**

配置的过程大致如下：

1. **在公司电脑上安装 Python 环境**：由于一些 **skill**（例如写 PPT、文档处理等）需要在客户端进行操作，因此需要在公司电脑上安装相应的 Python 环境。
2. **安装 skill**：在客户端安装所需的 **skill**，以便获取操作权限。

## **理想与现实的碰撞**

在配置过程中，我开始思考：为什么查询数据也需要在客户端进行呢？理论上，查询数据完全可以由服务端完成，分析后再将结果返回给客户端。这样一来，客户端只需要接收数据并展示即可，无需进行复杂的数据处理。

### **具体问题**

- **服务端 vs 客户端操作**：我原本设想的是，服务端负责查询数据并进行分析，然后将结果返回给客户端，客户端只需打开浏览器查看数据即可。然而，实际操作中，**OpenClaw** 却选择了在客户端模拟用户操作，打开浏览器查看数据。这让我不禁感叹：“你丫是个程序呀，模拟自己是个人看浏览器[呲牙][呲牙][呲牙]”。

### **对比与反思**

在反思过程中，我想到隔壁团队的做法：他们直接使用 **fetch** 命令进行数据查询，没有多余的客户端操作。这让我深刻体会到：**没有对比就没有伤害**。他们的方法显然更加高效和简洁。

## **改进建议**

经过这次部署，我总结了一些改进建议：

1. **优化数据查询流程**：尽量将数据查询和分析放在服务端完成，减少客户端的负担。
2. **简化客户端操作**：如果必须使用客户端进行某些操作，尽量简化操作步骤，提高效率。
3. **增强用户体验**：在设计工具时，多考虑用户的使用场景和需求，提供更加人性化的操作方式。

## **总结**

总的来说，这次 **OpenClaw** 的部署让我对工具的使用和配置有了更深的理解。虽然过程中遇到了一些问题，但通过对比和反思，我找到了改进的方向。希望我的经验能对大家有所帮助，也期待 **OpenClaw** 在未来的版本中能够更加完善。
